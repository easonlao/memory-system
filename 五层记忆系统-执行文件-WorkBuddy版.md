### 📁 第三步：路径配置（多端协同前提）

> [!important] 🔑 **设计前提：多端协同，不是单机专属**
>
> 本系统的核心设计目标是：**你可能在多个 AI 平台同时使用（WorkBuddy / Trae / Qclaw 等）**。
> 记忆数据不能锁死在任何一个平台的原生目录里，否则换平台=丢记忆。

**你需要确认两类路径：**

#### 第一类：公共文件夹（★ 多端共享，最重要的部分）

| 目录 | 用途 | 说明 |
|------|------|------|
| `~/[记忆共享中心]/` | 记忆数据主仓库 | 所有平台共用的唯一真实数据源 |
| `~/[记忆共享中心]/核心层/CORE.md` | L1 核心层 | 身份+价值观+底线 |
| `~/[记忆共享中心]/记忆规则/` | L2/L3 规则 | 认知层+行为层详细规则 |
| `~/[记忆共享中心]/SHADOW.md` | L5 潜意识层 | 言行偏差记录 |
| `~/[记忆共享中心]/技能配置/` | Skill 主版本 | 所有 Skill 的原始存放地 |

> 💡 **关于 `[记忆共享中心]`**
>
> 这是你的公共文件夹目录名占位符。你可以叫它 `ai-brain`、`记忆库`、`AI档案` 等——只要所有文件统一就行。**建议第一次搭建时就想好名字**，后续改名成本极高（需同步修改系统常量、软链接、所有路径引用）。

#### 第二类：WorkBuddy 原生路径（仅 WorkBuddy 使用）

| 目录 | 用途 | 说明 |
|------|------|------|
| `~/.workbuddy/rules/` | `.mdc` 协作契约 | 从公共文件夹同步过来的规则（自动注入！） |
| `~/.workbuddy/skills/` | Skill 软链接 | 指向公共文件夹的 Skill 主版本 |
| `{工作区}/.workbuddy/memory/` | 项目级工作记忆 | MEMORY.md + 每日日志 |

> 📍 **关键理解**：
> - `~/[记忆共享中心]/` = **数据源头**（你手动维护 / AI 写入）
> - `~/.workbuddy/rules/*.mdc` = **自动注入副本**（WorkBuddy 自动读取）
> - 两者的同步方式：**软链接** 或 **手动复制**
> - 如果只用 WorkBuddy 一个平台，可以把两者设为同一个目录；但**不建议**——因为以后加第二平台时又要迁移

### WorkBuddy 加载机制说明

> [!important] 理解这些机制是正确使用记忆系统的前提

#### 公共文件夹 vs WorkBuddy 原生：数据流向

```
[记忆共享中心]/CORE.md ──写入──→ ~/.workbuddy/rules/core-identity.mdc ──注入──→ AI 上下文
[记忆共享中心]/用户规则.md ──同步──→ ~/.workbuddy/rules/*.mdc ──注入──→ AI 上下文
[记忆共享中心]/技能配置/SKILL.md ──软链──→ ~/.workbuddy/skills/*/SKILL.md ──按需加载
{工作区}/.workbuddy/memory/MEMORY.md ──────────────────────────────→ 注入项目上下文
```

#### 加载机制表

| 文件类型 | 存放位置 | 加载时机 | 加载方式 | 多端可见？ |
|---------|---------|---------|---------|-----------|
| `IDENTITY.md` / `USER.md` | `~/.workbuddy/`（WB原生） | **每次对话自动** | system prompt 内置 | ❌ 仅WB |
| `rules/*.mdc` | `~/.workbuddy/rules/`（从公共同步） | **每次对话自动** | system prompt 内置 | ❌ 仅WB |
| `.workbuddy/memory/MEMORY.md` | 项目级 WB 原生 | **自动注入** | project context | ❌ 仅当前项目 |
| **公共文件夹文件** | `[记忆共享中心]/` | **AI 按需读取** | 工具调用 | ✅ **所有平台** |
| `skills/*/SKILL.md` | 软链接→公共文件夹 | 触发词匹配 | use_skill | ✅ **所有平台** |

> 💡 **核心认知**
>
> - **rules/*.mdc 是"缓存"**：真正的数据在 `[记忆共享中心]`，`.mdc` 是为了让 WorkBuddy 能自动注入而生成的副本
> - **Skill 通过软链接实现多端共用**：改主版本，三端同时更新
> - **MEMORY.md 和日志是项目级隔离的**：不同工作空间有不同的上下文，不适合共享

## 🔄 通用规则（v4.0新增）

### G1. Q编号脱钩原则

> [!warning] 记忆系统版本兼容性

- **禁止**在对话中使用具体题号（如"Q14-B"、"Q27的答案"）
- **必须**使用语义描述（如"你在失败时的归因模式"、"你的能量管理策略"）
- **原因**：问卷版本会迭代（33题→更少/更多题），题号引用会失效
- **例外**：报告模板内部的归属追踪表除外

### G2. 记忆架构完整加载

> [!success] 对话启动标准流程

每次开始需要深度了解你的任务时，AI 应按以下顺序确认记忆已加载：

```
# 第一阶段：身份（system prompt 自动注入 ✅ 无需手动操作）
1. IDENTITY.md  — 我是谁（自动注入）
2. USER.md      — 服务谁（自动注入）
3. rules/*.mdc  — 协作规则（自动注入）

# 第二阶段：情境（project context 自动注入 ✅ 无需手动操作）
4. MEMORY.md    — 当前目标/卡点/TTL状态（自动注入）

# 第三阶段：按需加载（AI 根据任务判断是否需要）
5. SHADOW.md    — 潜意识层（情绪复盘/言行不一致时）
6. 当日+昨日日记（回顾/规划/写日记时）
7. 相关 SKILL  （执行特定任务时，如日记/周报/读书笔记）
```

**为什么前四层不需要手动读取**：WorkBuddy 会自动把 IDENTITY.md、USER.md、rules/*.mdc、MEMORY.md 注入到每次对话中。你不需要手动告诉 AI 去读这些文件。

**什么时候需要手动加载**：
- 写今日日记时 → AI 应主动读取昨日日记和 SHADOW.md
- 做周总结时 → AI 应主动读取本周所有日志
- 做月度报告时 → AI 应主动读取本月所有日志 + SHADOW.md
- 执行读书笔记时 → AI 应加载读书助手 Skill

### G3. 注入路由规则

> [!note] 何时深入读取哪个记忆层

虽然前四层会自动注入基础信息，但某些任务需要**更深层的详细内容**：

| 触发场景 | 额外读取文件 | 原因 |
|---------|------------|------|
| 排计划/晨间规划 | 昨日日记 + MEMORY.md | 需要昨天的实际数据和当前目标 |
| 项目决策/业务讨论 | MEMORY.md 完整内容 | 需要当前目标和卡点详情 |
| 复盘/总结/周报 | 本周/本月所有日志 | 需要原始数据汇总 |
| 情绪低落/冲突 | SHADOW.md | 需要偏差追踪和对质偏好 |
| 写日记 | 昨日日记 + SHADOW.md | 需要 SHADOW_AUDIT 对照 |
| 读书笔记 | 读书助手 Skill | 需要笔记规范模板 |

> [!note] 复盘分层说明
>
> | 复盘类型 | 频率 | 主数据源 | 更新目标 |
> |---------|------|---------|---------|
> | **周报** | 每周 | **本周日志（PATH_DIARY）** | 提取本周数据 → **更新 L4（MEMORY.md）** |
> | **月度总结** | 每月 | **本月日志** | 提取洞察 → **更新 L2（用户基本规则.md）+ L4（MEMORY.md）+ L5（SHADOW.md）** |
> | **深度复盘** | 按需 | 全部记忆层 | 认知升级 → 可能更新任意层 |

#### G3b. 路由落地：信号词路由表

> [!success] 让路由从"概念"变成"可执行规则"
>
> G3 上面描述的是路由的**设计原理**——告诉 AI "什么场景该读什么"。
> 但光有原理不够：AI 可能"理解了但忘了执行"。
> **G3b 解决的是执行问题**——把路由变成 rules 文件，每次会话自动注入，强制执行。

**核心思路**：创建 `~/.workbuddy/rules/记忆路由表.mdc`，把路由逻辑写成信号词匹配规则。

**为什么需要这个**：

| 没有路由表 | 有路由表 |
|-----------|---------|
| AI 要"理解场景"才能判断读什么 | AI 只需匹配关键词，判断门槛低 |
| 靠 AI 自觉，跳过了无感知 | rules 强制注入，跳过了有标记可查 |
| 不同对话中行为不一致 | 每次会话规则一致 |

**路由表模板**（初始化时根据用户实际情况调整信号词）：

```markdown
# 记忆路由表 — 信号词匹配规则

> ⚠️ 强制规则：每次回复前，扫描用户消息是否匹配以下信号词。
> 匹配到 → 立即执行对应动作 → 基于读取内容回复。
> 不匹配 → 跳过，正常回复。

---

## 路由表

### 工具类路由（触发 Skill）

| 信号词（出现任一即触发） | 触发目标 |
|------------------------|---------|
| 写日记 / 今天的日记 / 日记复盘 | 每日伙伴 Skill |
| 写周报 / 这周总结 / 这周复盘 | 每日伙伴 Skill |
| 月度蒸馏 / 月报 / 月度校准 | 执行月度蒸馏规则 |
| 提到具体书名 / 读后感 / 这本书 | 读书助手 Skill |

### 记忆文件路由（读取对应文件）

| 信号词（出现任一即触发） | 读取文件 | 备注 |
|------------------------|---------|------|
| 计划 / 今天做啥 / 明天 / 安排 | `behavior-rules.mdc` 行为层 | 行为模式 |
| 要不要 / 选A还是B / 决策 | `core-identity.mdc` 或 MEMORY.md | 情境：目标+卡点 |
| 烦 / 累 / 压力大 / 内耗 | SHADOW.md | 最近偏差记录 |
| 复盘 / 总结 / 反思 / 回顾（无时间词） | 哲学思考框架（如有） | 认知层面的反思 |

### 兜底规则

- 复盘/总结 + 时间词（今天/这周/本月）→ 每日伙伴 Skill（工具类优先）
- 复盘/总结 + 无时间词 → 哲学思考框架（认知类兜底）
- 情绪词 + 否定词（不烦了/没觉得累）→ 不触发

---

## 执行规则

1. **匹配优先级**：一条消息匹配多个信号 → 全部执行，不只执行一个
2. **否定词排除**：信号词前有"不/没/别"时，不触发
3. **标记执行**：每次回复末尾追加一行路由状态标记（不要解释，只标记）：
   - 无匹配 → `[🧠 路由: 无匹配]`
   - 匹配1个 → `[🧠 路由: 行为层]` 或 `[🧠 路由: SHADOW]` 等
   - 匹配多个 → `[🧠 路由: 行为层+SHADOW]`
   - 触发 Skill → `[🧠 路由: 每日伙伴]` 或 `[🧠 路由: 读书助手]`
4. **读取路径**：`~/[记忆共享中心]/记忆规则/` 下的对应文件
5. **Skill 触发**：用 `use_skill` 调用对应 Skill
```

> [!tip] 关键设计决策
>
> 1. **信号词优于场景判断**：AI 不需要"理解用户在做什么"，只需匹配关键词——降低执行门槛
> 2. **否定词排除**：避免"不想做计划"误触发计划路由
> 3. **复盘歧义拆分**：复盘+时间词 = 工作复盘（Skill），复盘无时间词 = 认知反思（哲学框架）
> 4. **标记执行**：每次回复末尾一行标记，让路由可观测——漏了用户一眼就能看到
> 5. **信号词可自定义**：以上是默认模板，用户可根据自己的使用习惯调整信号词

> [!warning] 注意事项
>
> - 如果用户没有安装每日伙伴/读书助手 Skill，对应的工具类路由行应删除或注释
> - 如果用户没有 SHADOW.md 或哲学思考框架，对应行应删除
> - **路由表是用户专属的**，不同用户的信号词应不同——初始化时根据问卷答案调整

### G4. TTL 半衰期规则

> [!tip] 记忆有效期管理

| 记忆类型 | 半衰期 | 处理方式 |
|----------|--------|--------|
| 临时状态 | 1-3天 | 过期自动删除，不写入 MEMORY |
| 短期观察 | 7-14天 | 过期前确认"还成立吗？" |
| 中期经验 | 1-3个月 | 定期复盘，更新或归档 |
| 长期认知 | 6-12个月 | 年度大审查，写入 core-identity.mdc |
| 永久原则 | 永不过期 | 极少，仅 core-identity.mdc |

### G5. 每月蒸馏规则

> [!important] 月度记忆整理与分层写入

- **触发**：每月初（你可以设置 WorkBuddy automation 定时提醒，也可以手动触发）
- **执行**：对 WorkBuddy 说"帮我做月度记忆蒸馏"→ AI 读取当月所有日志，按以下四步处理：

| 步骤 | 做什么 | 写入目标 |
|------|--------|---------|
| ① 清理临时数据 | 删除情绪波动、待办进度等过时内容 | 从 MEMORY.md 中移除 |
| ② 保留永久记忆 | 提取原则、决策、教训等长期有价值的洞察 | 合并入 MEMORY.md |
| ③ 提炼行为模式 | 本月是否出现新的行为习惯或决策模式变化？ | 更新 L2（用户基本规则.md）|
| ④ SHADOW 月度统计 | 本月新增矛盾数、分类占比、未解决遗留数 | 更新 L5（SHADOW.md 末尾） |

- **体积控制**：MEMORY.md 建议不超过 150 行

> [!tip] 关于 Skill 用户
>
> 如果你有每日伙伴等 Skill，Skill 会做更细致的月报联动处理——**与 G5 不冲突，互为补充**。G5 始终是兜底保障。

#### 触发方式

| 方式 | 操作 | 适用场景 |
|------|------|---------|
| **手动触发（推荐）** | 说 **"执行月度校准"** 或 **"月度蒸馏"** | ✅ 所有用户，尤其是新人 |
| **自动触发（automation）** | WorkBuddy 中创建 monthly automation | 已配置自动化环境的用户 |
| **问卷触发** | 完成报告模板第八章「月度校准提示」5 问后自动执行 | 做完问卷的用户 |

> [!tip] 新人推荐
>
> 直接说 **"执行月度校准"** 即可。AI 会读取当月日志，按上述 4 个步骤逐项处理并写入对应文件。不需要配 automation，不需要装任何额外工具。

### G5b. 周报更新规则

> [!success] 每周情境层刷新（L4 维护）
>
> **目的**：L4 情境层（目标/卡点/需求）变化频率高，每月更新一次太慢。周报是 L4 的自然更新节点。
>
> **触发方式**：用户说 **"写周报"**、**"周复盘"**、**"本周总结"**

#### 周报数据源与提取内容

| 数据来源 | 提取内容 | 写入目标 |
|---------|---------|---------|
| 本周日志（PATH_DIARY） | 本周完成事项、时间分配、能量状态 | **MEMORY.md（L4）** — 更新"当前状态"段 |
| 上周计划 vs 实际完成 | 计划偏差、未完成原因 | **MEMORY.md（L4）** — 更新"阻力"段 |
| 90天核心目标进展 | 目标推进百分比、里程碑达成情况 | **MEMORY.md（L4）** — 更新"当前目标"段 |

#### 执行流程

```
1. AI 读取当周所有日志
2. 提取上述三类信息
3. 与 MEMORY.md 中现有内容对比
4. 标注变化项，更新 MEMORY.md
5. 输出简要周报摘要给用户确认
```

#### 与月度蒸馏的区别

| 维度 | 周报（G5b） | 月度蒸馏（G5） |
|------|-----------|--------------|
| **频率** | 每周 | 每月 |
| **深度** | 浅层：只更新 L4 状态数据 | 深层：提取洞察，可能更新 L1-L5 |
| **数据范围** | 当周日志 | 当月全部日志 |
| **认知提炼** | 不做 | ✅ 提取策略/教训写入 L2 |
| **触发词** | "写周报"/"周复盘" | "执行月度校准"/"月度蒸馏" |

> [!note] 新人提示
>
> 周报不是必须的。如果你的目标/卡点每周都在变，建议每周做一次。如果变化不大，可以跳过，等月度蒸馏一起处理。

### G6. 规则文件管理

> [!success] .mdc 规则文件的编写和使用

WorkBuddy 的规则通过 `~/.workbuddy/rules/` 下的 `.mdc` 文件实现。

**规则文件的特点**：
- 放入 `rules/` 目录的 `.mdc` 文件会在**每次对话启动时自动注入 system prompt**
- 不需要任何额外配置，创建即生效
- 支持多个文件，全部都会被注入

**建议的规则文件拆分**：

| 文件名 | 内容 | 预估行数 |
|--------|------|---------|
| `core-identity.mdc` | L1 核心层（价值观/红线）+ L2 认知层（交互风格/决策画像） | ~150 行 |
| `behavior-rules.mdc` | L3 行为层（行为模式/能量管理）+ 协作契约 | ~150 行 |

**规则编写规范**：
```markdown
# 规则标题

## 元信息
- 版本：v1.0
- 更新日期：YYYY-MM-DD
- 对应层级：L1/L2/L3

## 具体规则

（用清晰的条目列出每条规则）

---
```

**注意事项**：
- 规则总行数建议控制在 **300 行以内**（太多会影响 system prompt 空间）
- 只放"铁律"级内容，不放临时性或细节性内容
- 修改规则文件后，**下次新对话立即生效**（不需要重启）

### G7. Skill 管理规则

> [!success] Skill 技能模块管理

Skill 是 WorkBuddy 的标准化工作流模块，存放在 `~/.workbuddy/skills/{名称}/SKILL.md`。

**何时创建新 Skill**：
当你发现自己在不同对话中反复描述同样的任务流程时，就应该把它做成 Skill。

**Skill 的加载机制**：
- Skill **不会**自动注入（不像 rules/*.mdc）
- Skill 通过**触发词匹配**或 AI **主动判断**来加载
- 你也可以在对话说"使用 XX Skill"来显式触发

**常用 Skill 建议**：

| Skill 名称 | 触发词 | 用途 |
|-----------|--------|------|
| daily-diary | 日记/每日记录/今天复盘 | 每日日记标准化输出 |
| weekly-review | 周报/周总结/本周回顾 | 周度复盘和规划 |
| reading-notes | 读书笔记/阅读记录 | 结构化读书笔记 |
| project-review | 项目复盘/项目总结 | 项目级别复盘 |

**Skill 模板结构**：
```markdown
# {Skill 名称}

## 触发条件
（什么情况下启用这个 Skill）

## 输入
（需要用户提供什么信息）

## 执行步骤
Step 1: ...
Step 2: ...
Step 3: ...

## 输出格式
（结果应该长什么样）

## 质量检查清单
- [ ] 检查项 1
- [ ] 检查项 2
```

### G8. 成长箱晋升机制

> [!success] 错误晋升流程

当同一类问题出现 3 次时，AI 应主动提醒将其晋升为规则。

**晋升标准**：
- 同一类错误出现 **3 次** → 准备晋升为规则
- AI 主动提醒用户确认 → 用户同意后写入对应的 `.mdc` 文件

**晋升流程**：
1. **累计计数**：在对话中记录同类错误的出现
2. **达到阈值**：第 3 次发生时，AI 提醒："这个问题已经出现 3 次了，建议写进规则"
3. **规则提炼**：从错误中提炼出通用规则表述
4. **写入规则**：追加到对应的 `core-identity.mdc` 或 `behavior-rules.mdc`
5. **通知用户**：告知用户写了什么规则、写到哪里

### G9. 自定义层管理规则

> [!success] 自定义记忆层指南

自定义层是五层记忆架构的扩展，用于存储五层无法覆盖的领域专精、项目特定、阶段性目标等内容。

#### 何时需要自定义层
- **领域专精**：特定领域的知识、技巧、偏好
- **项目特定**：重要项目的架构、进度、决策
- **阶段性目标**：有明确截止日期的目标
- **持续跟进**：需要长期追踪但不属于五层记忆范畴的内容

#### 自定义层结构模板
每个自定义层应包含以下结构：

```markdown
# 自定义层名称

## 索引
| 优先级 | 项目/目标 | 状态 | 最后更新 |
|--------|----------|------|----------|
| P1 | 项目A | 进行中 | YYYY-MM-DD |
| P2 | 项目B | 计划中 | YYYY-MM-DD |

## 状态定义
- 计划中：尚未开始
- 进行中：正在执行
- 暂停中：暂时停止
- 已完成：已达成
- 已归档：已完成但保留记录

## 详情
### 项目A
[详细描述、任务分解、进展记录]
```

#### 管理规则
1. **按需创建**：不要预先创建大量自定义层，根据实际需要创建
2. **定期更新**：索引文件每周更新，确保状态准确
3. **TTL设置**：阶段性目标设置明确截止日期，到期自动评估是否归档
4. **联动机制**：项目进展可与周报、月报联动，形成闭环管理
5. **避免重复**：内容已在五层记忆中的（如情境层的目标），不要在自定义层重复存储

#### 加载方式
- **按需加载**：自定义层非启动必读，通过触发词或上下文自动加载
- **示例触发词**："查看我的项目跟进"、"更新XX目标"
- **优先级**：高于基础记忆层，低于核心层

#### 通用模板

> 可复制以下模板创建新的自定义层：

```markdown

> 创建日期：YYYY-MM-DD
> 最后更新：YYYY-MM-DD

- 已完成：已达成
- 已归档：已完成但保留记录

## 项目/目标详情

### [项目1名称]

- 优先级：P1

[具体目标]

### G10. 题目库管理规则

> [!success] 题目管理机制

**题目变更流程**：
1. 所有 33 道题目的最新版本就在本文档的「📝 用户问卷」部分

### G11. 记忆文件版本管理

> [!success] 版本控制机制

建立记忆文件的版本控制机制，确保记忆系统的可追溯性和稳定性。

#### 版本管理方法
- **文件命名**：在文件名后添加版本号，如 `CORE.md.v1`
- **变更记录**：在文件末尾添加变更记录，记录每次修改的内容和日期
- **备份机制**：定期备份记忆文件，防止数据丢失
- **回滚机制**：当新版本出现问题时，能够快速回滚到之前的版本

#### 版本控制流程
1. **修改前备份**：在修改记忆文件前，先创建备份
2. **版本号更新**：修改完成后，更新文件版本号
3. **变更记录**：在文件末尾添加变更记录
4. **验证测试**：测试修改后的记忆文件是否正常工作

### 🌅 工作流 A：早晨规划 (Trigger: "早上好" / "开始规划")

### 🔒 安全哲学：Fail-Closed 默认假设

> **核心信念**：AI会犯错。路径会变。记忆会过时。格式会漂移。
>
> 因此本Skill的**所有默认值都按"最坏情况"设计**：
> - 默认假设：路径已变更 → 必须验证（读 config.yaml）
> - 默认假设：记忆不可靠 → 必须读取实际文件
> - 默认假设：上次执行成功 → 必须重新检查
> - 默认假设：格式没漂移 → 必须对照模板（读 references/note-template.md）
>
> **这不是不信任，这是工程化思维。** 禁令 > 指令，验证 > 信任，实际文件 > 记忆。

**此检查清单为强制入口，任何时候不得跳过。违反以下任一项，视为系统性故障，必须删除当前笔记重新开始。**

```

□ 已制定执行步骤，确保不违反铁律
□ 执行过程中每步核对铁律
□ 完成后自查是否违反铁律
```

### 🔒 配置验证铁律

**绝对禁止**：
- ❌ 禁止只读 skill 主文档，不读 `config.yaml`
- ❌ 禁止发现路径不一致时自由发挥判断
- ❌ 禁止未向用户确认就执行配置变更
- ❌ 禁止相信文档而不验证实际文件位置

**必须执行**：
- ✅ 必须读取 `skill-读书助手.md` 全文
- ✅ 必须读取 `config.yaml` 确认路径配置
- ✅ 必须检查现有笔记实际存放路径
- ✅ 必须对比 config.yaml 路径 vs 实际路径是否一致
- ✅ 发现不一致必须立即停止，向用户确认
- ✅ 必须显式输出检查清单完成情况，让用户监督

**配置验证流程**（每次任务前逐项输出）：
```
1️⃣ 读取 skill 主文档 → 输出："已读取 skill-读书助手.md"
2️⃣ 读取 config.yaml → 输出："路径配置：XXX"
3️⃣ 检查实际文件位置 → 输出："现有笔记存放在：XXX"
4️⃣ 对比一致性 → 输出："config vs 实际：一致/不一致"
5️⃣ 如不一致 → 停止执行，向用户确认
6️⃣ 获得确认后 → 方可执行
```

## Overview

本skill帮助用户完整阅读书籍并生成高质量的结构化读书笔记。核心原则是**必须完整阅读书籍**，确保所有内容准确可靠。

## 工作流程（4阶段）

### 阶段1：读取阶段

1. **识别书籍文件格式** — PDF/EPUB/TXT/Markdown，确认路径和可访问性
2. **读取现有笔记格式（必做）** — 读至少2本现有笔记，熟悉标准格式
3. **完整阅读书籍** — 从头到尾完整读取，包括封面、目录、前言、正文、后记

### 阶段2：理解阶段

1. **把握整体框架** — 核心主题、主要论点、章节关系
2. **识别关键要素** — 重要概念、核心论据、关键案例和数据

### 阶段3：提取与整理

按9模块结构提取内容。**详细模板见 `references/note-template.md`**。

### 阶段4：质量检查

**按 `references/quality-checklist.md` 执行三步检查**。
